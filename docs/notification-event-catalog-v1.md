# Notification Event Catalog v1

## Scope

SLA monitoring events generated by the cycle `POST /api/sla/run` and by background SLA workers.

## Event types

### 1. `SLA_WARNING`

Trigger:

- due date is inside warning horizon and not overdue.

Severity:

- `Warning`.

Sources:

- procedure proposal due date;
- procedure required subcontractor deadline;
- contract end date;
- contract milestone planned date.

Delivery rule:

- one notification per unique key `(EntityType, EntityId, DueDate, Severity)` while event stays open.

### 2. `SLA_OVERDUE`

Trigger:

- due date is in the past (`DueDate < UtcToday`).

Severity:

- `Overdue`.

Sources:

- same as warning.

Delivery rule:

- one notification per unique key `(EntityType, EntityId, DueDate, Severity)` while event stays open.
- warning and overdue have separate keys, so overdue escalation is sent even if warning was already sent.

## Recipient resolution

Current routing baseline:

- recipient is `ResponsibleCommercialUser.Email` derived from linked procedure;
- for milestones/contracts, recipient is resolved through contract -> procedure -> responsible user;
- if recipient email is empty, event stays open and `LastNotificationError` is populated.

## Notification state tracking

Fields in `SlaViolation`:

- `NotificationAttempts`;
- `LastNotificationAttemptAtUtc`;
- `LastNotificationSentAtUtc`;
- `LastNotificationError`.

## Event closure

Open event is closed (`IsResolved=true`) when its condition is no longer detected in the current cycle.

Example:

- milestone progress becomes `100%` -> corresponding milestone warning/overdue event is resolved.
